137 * Правила именования переменных/функций/классов/файлов
138 * ----------------------------------------------------
139 *
140 * - Имена переменных должны быть существительными, функций -
141 * глаголами/командами.
142 * - Имена файлов должны совпадать с именами объявленных в них классов (включая
143 * пространство имен).
144 * - \ref IdentifiersAnchor "Правила именования идентификаторов".
145 *
146 * Пространства имен
147 * -----------------
148 *
149 * - Каждый отдельный модуль программы должен иметь свое пространство имен.
150 * - Не использовать директиву using namespace вне функций.
151 * - Пространство имен должно завершаться комментарием "} // namespace <name>".
152 * - Код внутри пространства имен должен идти по левому краю.
153 * - Вместо статических констант использовать константы в не именованных
154 * пространствах имен (использовать их в заголовочных файлах только для
155 * объявления констант!).
156 *
157 * Конструкторы/деструкторы
158 * ------------------------
159 *
160 * - Всегда явным образом определять конструктор по умолчанию, инициирующий
161 * внутренние переменные класса.
162 * - Использовать ключевое слово explicit для конструкторов с одним аргументом.
163 * - Если класс не имеет явного конструктора копирования, наследовать его от
164 * boost::noncopyable.
165 * - Деструктору всегда явным образом указывать noexcept.
166 *

237 * Под вопросом
238 * ============
239 *
240 * - Запятые в списке инициализации переменных в конструкторе ставить после
241 * переменной.
242 * - Вместо \<iostream\> в заголовочных файлах писать \<iosfwd\> для ускорения
243 * компиляции.
244 * - Для быстрого копирования потоков ввода - вывода(из одного в другой),
245 * использовать функцию потока .rdbuf() (cout << file.rdbuf()).
246 * - С помощью функции std::uncaught_exception можно проверить, как завершает
247 * работу функция: это нормальный возврат или было выброшено исключение.
248 * - Преобразовывайте тип числа с плавающей точкой в целочисленный тип
249 * исключительно через функции стандартной библиотеки(«round», «floor», «ceil»
250 * и т.п.)
251 * - Для инициализации переменных использовать списки инициализации (они не
252 * позволяют сужение типов: int x1 = { 7.3 }; //compilation error).
253 * - Профайлер не может подсказать, какие inline функции не должны быть
254 * встраиваемыми!
255 * - Строки хранить в отдельном файле (лучше.cpp).
256 * - Исключение не должно покидать деструктор (особенно это касается работы
257 * с контейнерами STL).
258 * - Для класса предоставлять функцию swap() (она должна быть бессбойной!),
259 * которую можно использовать для присваивания со строгими гарантиями
260 * безопасности (для примитивных типов использовать std::swap())
261 * \code
262 
263 T & T::operator= (T Temp) // По значению!
264 {
265  swap(Temp);
266  return *this;
267 }
268 
269 * \endcode
270 * - Для отсортированного набора объектов можно использовать индексные
271 * контейнеры, содержащие итераторы на основной контейнер.
272 * - Для добавления элементов в контейнер предпочтительнее использовать операции
273 * с диапазонами.
274 * - container<T>(c).swap(c); // Удаление всей лишней (зарезервированной)
275 * емкости контейнера.
276 * - container<T>().swap(c); // Удаление всех элементов и очистка памяти
277 * контейнера.
278 * - find / find_if и count / count_if - поиск в неотсортированном диапазоне.
279 * - lower_bound, upper_bound, equal_range(реже binary_search) - поиск
280 * в отсортированном диапазоне.
281 * - Функциональные объекты должны содержать константный operator().
282 * - Побочные подзадачи выделять в отдельный код.
283 */
