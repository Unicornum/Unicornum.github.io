Documentation

Проект утилиты предварительной обработки файлов решения.

Предназначен для:

  • Обработки файла лога сборки решения скриптом.
  • Обработки файла покрытия кода тестами.
  • Обработки файла расчета метрик кода в решении.
  • Генерации документации.

Использование утилиты

  • Подключить к решению проект Documentation из папки Externals\Tools\Documentation.
  • Одному из проектов решения установить зависимость от проекта Documentation и в качестве события после построения добавить запуск Documentation.exe (в папке $(SolutionDir)$(Configuration)\Bin\x64) с нужными командами.

Вспомогательные параметры

  • –file=path - путь к исходному файлу (трактуется в зависимости от указанной команды).
  • –path-to-solution=path - полный путь к папке решения, которое обрабатывается (используется совместно с командой –process-report).
  • –max-records - максимальное количество записей в отчете (при добавлении новых записей самые старые будут удаляться; используется совместно с командой –process-report).
  • –platform - текущая собираемая платформа (используется совместно с командой –make).

Команды

Заметки
Команды исключают использование друг друга.
  • –help - выводит в консоль список доступных команд.
  • –process-report=type - обработка файла, заданного как параметр –file (также требуется параметр –path-to-solution):
    • build - обработка файла лога сборки скриптом (в той же папке и с тем же именем, но с расширением .html генерируется файл, группирующий и подсвечивающий цветом ошибки и предупреждения с разбиением оных на группы по проектам).
    • coverage - обработка файла отчета покрытия кода тестами (из xml файла отчета генерируется html файлы, пригодные для включения в документацию Doxygen).
    • metrics - обработка результата вычисления метрик кода в решении (–file задает путь к папке, где располагаются файлы отчета SourceMonitor, –max-records задает количество проблемных мест в коде, которое будет выводиться для каждой метрики).
  • –make - команда построения документации (например, 'doxygen.exe PathToConfigFile% > PathToReport'), требуются параметры:
    • –file - путь к файлу версии решения.
    • –platform - собираемая платформа.

Localization

Проект утилиты генерации заголовочного файла с идентификаторами строк, используемыми для получения у модуля локализации программы текста выбранного языка программы.

  • Утилита генерирует заголовочный файл из xml файла, переданного в качестве параметра.
  • Значение узла path будет интерпретироваться как путь к папке расположения файлов локализации программы (относительно корневой папки расположения exe файла программы).
  • Узлы, не содержащие значений будут интерпретироваться как указание создать пространство имен с именем узла.
  • Узлы, содержащие значения, будут интерпретироваться как строки, которые будет использовать программа (для них будут сгенерированы идентификаторы).
  • При обработке переданного файла утилита также будет сравнивать с указанным файлом все xml файлы в той же папке с выводом найденных расхождений в консоль.

Resource

Проект утилиты генерации заголовочного файла, содержащего бинарные данные указанного файла. Назначение - хранение непосредственно в программе ресурсов небольшого размера (иконок, курсоров и т.п.).

  • Если в качестве пути (–path="path/to/...") задан путь к файлу, заголовочный файл генерируется только для него, если путь к папке - то для всех файлов в этой папке, исключая файлы .hpp.
  • Сгенерированный заголовочный файл будет располагаться в той же папке, что и исходный файл и будет иметь то же имя (с расширением .auto.hpp).
  • Имя массива данных в созданном файле будет совпадать с именем исходного файла (учитывать при именовании файла!).

Использование утилиты

  • Подключить к решению проект Resource из Externals/Tools.
  • Создать проект служебной программы, в который добавить необходимые файлы ресурсов, добавить ему зависимость от проекта Resource и добавить в качестве события после сборки
    $(ResultDir)Resource.exe --path=$(ProjectDir)
    %(Command)

RocketRenderToFile

Использование утилиты

  • Добавить в решение проект Externals/Tools/RocketRenderToFile.
  • Добавить проекту тестирования зависимость от проекта RocketRenderToFile.
  • В тестах подготовить файлы, используемые в тестируемом .rml файле (стилей, изображений, шрифтов) с учетом их взаимного расположения.
  • Запустить с требуемыми параметрами утилиту "$(SolutionDir)/$(Configuration)/Bin/x64/RocketRenderToFile.exe", проверяя совпадение полученного скриншота с ожидаемым изображением.

Параметры командной строки

  • help - вывод в консоль описания параметров командной строки.
  • source-file - путь к тестируемому .rml файлу.
  • dest-file - путь к .png файлу, в который будет сохранен скриншот экрана для указанного .rml файла.
  • width - ширина генерируемого изображения (по умолчанию - 400).
  • height - высота генерируемого изображения (по умолчанию - 800).
  • bkcolor - цвет фона, поверх которого будет рендериться полученный .rml файл (по умолчанию - 0xE30B5C (малиновый)).
  • fonts-dir - путь к папке, из которой должны быть загружены .ttf шрифты, используемые в указанном .rml файле.

Version

Проект утилиты манипуляции версией решения.

Предназначен для:

  • Обновления номера ревизии в файле версии решения.
  • Увеличения на 1 указанного номера версии в файле версии решения.
  • Формирование заголовочного файла версии решения для включения в состав библиотеки.
  • Обновление ревизии файлов решения в файле AndroidManifest.xml.
    <manifest ...
    android:versionCode="CURRENT_REVISION">
    ...
  • Формирование xml файла версии решения для включения в Android package (должен располагаться в папке Packaging\res\values). Образец формируемого файла:
    <?xml version="1.0" encoding="utf-8"?>
    <resources>
    <string name="CompanyName">UnicornumVerum</string>
    <string name="FileDescription">Набор вспомогательных библиотек</string>
    <string name="FileVersion">3.8.0 Revision: 2831 Platform: x64 Build: 01:29 09 Сентябрь 2017</string>
    <string name="FileVersionShort">3.8.0.2831 x64</string>
    <string name="LegalCopyright">© UnicornumVerum 2007 - 2017 (unicornum.verum@gmail.ru)</string>
    <string name="ProductName">Alicorn</string>
    <string name="ApplicationName">Alicorn</string>
    <string name="ProductVersion">3.8.0 Revision: 2831 Platform: x64 Build: 01:29 09 Сентябрь 2017</string>
    <string name="ProductVersionYear">3.8.0.2017</string>
    <string name="ProductVersionMajor">3</string>
    <string name="ProductVersionMinor">8</string>
    <string name="ProductVersionService">0</string>
    <string name="ProductRevision">2831</string>
    </resources>
    Строки версий решения можно использовать следующим образом - в файле AndroidManifest.xml изменить:
    <manifest ...
    android:versionName="@string/ProductVersionYear"
    ...

Использование утилиты

  • Подключить к решению проект Version из папки Externals\Tools\Version.
  • Одному из проектов решения нужно установить зависимость от проекта Version и в качестве события после построения добавить запуск Version.exe (в папке $(SolutionDir)$(Configuration)\Bin\x64) с нужными командами.

Команды

Заметки
Если задано несколько команд, они будут выполнятьсяв указанном порядке.
  • –help - выводит в консоль список доступных команд (исключает все остальные команды).
  • –file=path - путь к файлу Version.hpp с версией решения (обязательный параметр для всех последующих команд).
  • –platform - собираемая платформа (x64, Win32 и т.п.).
  • –update - в файле версии решения следует обновить номер ревизии файлов решения, время/дату, а также установить переменную окружения VERSION_SOLUTION, содержащую версию решения без ревизии решения (если файл, указанный в –file не существует, от будет создан со значениями по умолчанию).
  • –increment=value - в файле версии решения следует увеличить на 1 указанный номер версии решения (service - увеличивает 0.0.x).
  • –set-environment-variables - в консоль выводятся команды установки переменных окружения, устанавливающие значения версии решения:
    • VERSION_SOLUTION - версия решения вида '1.2.3'.
    • VERSION_PRODUCT_NAME - название продукта вида 'Alicorn'.
  • –build-header-version=path - path задает путь к файлу, который должен быть перезаписан с учетом текущей версии решения.
  • –update-android-manifest=path ** - path задает путь к файлу AndroidManifest.xml, в котором следует установить текущий номер ревизии файлов решения.
  • –build-android-version=path - path задает путь к файлу версии Android package (version.auto.xml), который будет перезаписан с учетом версии решения.
Уст.:
  • –configuration - собираемая конфигурация (x64, Win32 и т.п.), вместо параметра использовать platform.

Использование переменных окружения

Т.к. при установке переменных окружения внутри процесса они будут доступны только самому процессу, утилита выводит в консоль набор команд, которые можно использовать в командном файле следующим образом:

for /f "usebackq delims=;" %%a in (`version.exe --set-environment-variables`) do (
%%a
)
echo VERSION_SOLUTION: %VERSION_SOLUTION%
